Pattern: Optional Database — Null Object

Instead of throwing an error when DATABASE_URL is missing, the app exports null and checks if (pool) before every DB call. This is a lightweight "null object" pattern — local dev needs zero DB setup, production gets persistence.

Why JSONB for movies? Each room stores the full 20-movie array as a single JSON blob. There's no need to query individual movies from Postgres — the whole array is read back in one go. JSONB is more storage-efficient than TEXT and supports GIN indexes if needed later.

Why Set for random pages?

A Set guarantees uniqueness with O(1) membership checks. The while loop keeps trying until it has 3 unique pages. Since max=10 and count=3, worst-case retries are low — this is efficient. Alternative: shuffle [1..max] and slice — cleaner but overkill for 3 picks from 10.

Why 3 random pages instead of page 1 every time?

TMDB page 1 of any genre is always the same most-popular films. Fetching 3 random pages (from the top 10) gives variety — two rooms with the same genre will likely get different movies. The trade-off: popularity sorting means pages 8-10 have lower-rated obscure films, so max=10 keeps quality reasonable.

Why hold matches until both players finish?

If matches were broadcast immediately when detected, Player A (who finished quickly) would know what Player B liked before B was done. This would be a spoiler — B might even change their votes strategically. The "hold until done" design preserves the mutual surprise. The match detection still runs in real time, but the reveal is deferred.

io.to() vs socket.to() vs socket.emit()

  • socket.emit('x', data) — sends only to THIS socket (the caller)
  • socket.to(room).emit('x', data) — sends to everyone in room EXCEPT this socket
  • io.to(room).emit('x', data) — sends to EVERYONE in room including this socket

The app uses all three: socket.emit for private responses (room-created, room-joined), socket.to for notifications to the other player (partner-joined from joiner's side), and io.to for broadcast-to-all (all-done, partner-disconnected).

Why a Singleton Socket?

JavaScript ES module imports are cached — the same module instance is returned every time. So import socket from '../socket.js' in Home.jsx and Room.jsx returns the same socket object. This means:

  • No need for React Context or global state to share the socket
  • Events registered in one component fire even when the component re-renders
  • The socket stays connected across route changes (Home → Room)

Dual Range Slider — Two Overlapping <input type="range">

There's no native HTML dual-handle slider. The trick: two <input type="range"> elements are absolutely positioned on top of each other with background: transparent. Only the thumb (the draggable circle) responds to interaction — the track is invisible. A separate <div> with a gold gradient is positioned using the computed percentages to visually show the selected range.

The pointer-events: none on the input and pointer-events: all on ::-webkit-slider-thumb ensures only the thumbs are interactive, not the underlying track.

The Stale Closure Trap — Why Refs are Used Here

Event listeners registered inside useEffect([], []) (empty deps) are created once and never re-created. They "close over" the initial values of voted and onVote. If state changes, the listener still references the old values — a "stale closure".

Solution: Use useRef. A ref is an object { current: value }. The listener closes over the ref object (which never changes), but reads ref.current at call time — getting the latest value.

Without this pattern: voting the first card would work, but the second card's listener would still think voted=false (the initial value), causing double-votes or broken state.

Why Deterministic Confetti?

If CONFETTI were generated inside the component body with Math.random(), every re-render would create new random positions. In React StrictMode (dev), components render twice — you'd see the confetti "jump" to different positions. By computing positions at module load time (outside the component), the positions are fixed for the entire session. This is a classic React pattern for stable animations.
