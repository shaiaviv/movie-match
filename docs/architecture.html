Architecture Overview

MovieMatch solves a two-sided coordination problem: two people who want to watch a movie together, but can't agree on one. The design philosophy is zero infrastructure — no accounts, no database required, no long-lived state. Two people create a temporary room, swipe through the same movies in real time, and the server only needs to survive long enough for one session.

The architecture separates into three clear layers: transport (Socket.io handles all real-time communication), state (roomManager.js owns all mutable room data), and presentation (React components handle all UI rendering with no shared global state library).

Component Map

server/index.js

Entry point · Express + Socket.io
  • Owns: HTTP server, socket event handlers, health endpoint
  • Imports: fetchMovies (tmdb.js), all room functions (roomManager.js), initSchema (db.js)
  • Exports: nothing (entry point)
  • Called by: Node.js runtime directly

server/roomManager.js

Business logic · Pure functions
  • Owns: rooms Map (all room state), all vote/match logic
  • Imports: pool from db.js
  • Exports: createRoom, joinRoom, getRoom, getRoomBySocket, recordVote, markDone, removeUser
  • Called by: index.js socket handlers

server/tmdb.js

External API · Side effects
  • Owns: TMDB URL building, page fetching, movie formatting
  • Imports: node-fetch
  • Exports: fetchMovies (used only by index.js)
  • Called by: index.js on create-room event

server/db.js

Optional persistence · Postgres
  • Owns: pg.Pool, schema creation
  • Imports: pg (node-postgres)
  • Exports: pool (null if no DATABASE_URL), initSchema
  • Called by: roomManager.js (pool), index.js (initSchema)

client/src/socket.js

Singleton · WebSocket client
  • Owns: the Socket.io client instance
  • Imports: socket.io-client
  • Exports: single socket instance (default)
  • Used by: Home.jsx and Room.jsx

client/src/pages/Home.jsx

Page · Filters + Room creation
  • Owns: all filter state, mode state, YearRangeSlider, Pills, CinemaParticles sub-components
  • Emits: create-room, join-room
  • Listens: room-created, room-joined, error

client/src/pages/Room.jsx

Page · Swiping screen
  • Owns: game state (index, voted, localDone, done, matches)

  • Listens: room-joined, partner-joined, all-done, partner-disconnected, error

client/src/components/MovieCard.jsx

  • Owns: drag state, animation state, event listeners

Complete Call Graph — create-room to all-done

Home.jsx handleCreate()
User clicks "Start" → emits create-room
Payload: { genreId, yearFrom, yearTo, minRating, runtimeMin, runtimeMax, language, certification } — undefined values omitted from JSON
→ No response yet. Socket registered. UI shows "Finding films…"
↓ socket.emit('create-room', payload)
index.js on('create-room')
Validates TMDB_API_KEY, then calls fetchMovies()
TMDB_API_KEY guard fires first. If missing, emits error and returns.
→ Calls fetchMovies(apiKey, genreId, opts) in tmdb.js
↓ await fetchMovies(apiKey, genreId, { yearFrom, ... })
tmdb.js fetchMovies()
Picks 3 random pages → builds TMDB Discover URL → fetches all 3 in parallel
Promise.all([fetch(page1), fetch(page2), fetch(page3)]) — fires simultaneously, ~200-400ms total
→ Returns movie[] (20 shuffled, formatted movies)
↓ movies: array of 20 movie objects returned to index.js
roomManager.js createRoom(socketId, movies)
Generates 6-char code → stores room in Map → optionally writes to Postgres
Room structure: { id, users:[socketId], movies, votes:{ socketId:{} }, matches:[], done:Set() }
→ Returns roomId string (e.g. "ABCDEF")
↓ socket.join(roomId) + socket.emit('room-created', { roomId, movies })
Home.jsx on('room-created')
Saves movies to sessionStorage → navigate('/room/ABCDEF')
sessionStorage key 'movies' = JSON.stringify(movies). Survives page refresh.
→ React Router navigates to Room.jsx
↓ React Router renders Room.jsx with :roomId = "ABCDEF"
Room.jsx useEffect (mount)
Connects socket if needed → emits join-room for registration → registers all socket listeners
Always emits join-room, even for the creator — ensures socket is registered with the room for votes
→ Server calls joinRoom() which finds creator already in room (idempotent), returns room
↓ Second player joins via URL or code share
roomManager.js joinRoom(code, socketId)
Finds room → validates capacity → adds joiner to users + votes
Also handles DB fallback: if room not in memory (server restart), queries Postgres
→ Returns { room } on success, { error } on failure
↓ socket.to(room.id).emit('partner-joined') + socket.emit('partner-joined')
MovieCard.jsx touch/mouse events → finish()
Detects drag end → computes delta and velocity → triggers exit or spring-back
Exit: setIsExiting(true), setDragX(±600) → setTimeout 350ms → calls onVote(liked)
→ Calls onVote(liked) which is handleVote in Room.jsx
↓ onVote(liked) → handleVote(liked) in Room.jsx
Room.jsx handleVote(liked)
Emits vote → advances index (or marks localDone if last movie)
If last movie: setLocalDone(true) + socket.emit('player-done', roomId)
→ socket.emit('vote', { roomId, movieId, liked })
↓ socket.emit('vote', { roomId, movieId: 550, liked: true })
roomManager.js recordVote()
Stores vote → if liked, checks if other player also liked → pushes to matches if so
Match detection logic: votes[other]?.[movieId] === true. Both must have liked it.
→ void. Matches accumulate in room.matches[]
↓ When both players finish: socket.emit('player-done') from each player
roomManager.js markDone()
Adds player to done Set → checks if all users are done → returns matches or null
room.users.every(id => room.done.has(id)) — true only when ALL players done
→ room.matches[] when all done, null while waiting
↓ if (matches !== null): io.to(roomId).emit('all-done', matches)
Room.jsx on('all-done')
setMatches(m) + setDone(true) → renders results screen with match list
Both players receive this simultaneously. UI shows matched movies or "no matches".
→ Final state. User can "Start Over" → clears sessionStorage, navigates to /

Technology Decisions

Socket.io
Real-time bidirectional events between server and clients. Automatically falls back to HTTP long-polling if WebSockets aren't available. The room-based broadcasting (io.to(roomId)) makes multi-player sync trivial.
Alternative: Native WebSocket API. Pros: no dependency, smaller bundle. Cons: no built-in rooms, no reconnection logic, no fallback — everything has to be built manually.
React 18 + Vite
React for component-based UI with hooks. Vite for near-instant HMR in dev and optimized ES module builds. No Next.js needed — this is a pure client-side SPA.
Alternative: Next.js. Unnecessary overhead — no SSR needed, no API routes (Socket.io runs separately), no RSC benefits here.
In-memory Map (rooms)
Zero setup, O(1) lookup, naturally scoped to the process lifetime. Rooms are ephemeral by design — they don't need to outlive the server process.
Alternative: Redis. Would enable horizontal scaling (multiple server instances) but adds significant operational complexity for a single-server app.
Tailwind CSS
Utility-first CSS — all styles colocated with markup. Custom design system (noir/gold/crimson/cream palettes) defined in tailwind.config.js. No CSS file to maintain per component.
Alternative: CSS Modules or styled-components. More isolation but more files. Tailwind's JIT purges unused styles so final bundle is small.
TMDB Discover API
Free, requires only a free account API key. Supports all needed filters (genre, year, runtime, language, certification) in a single URL. Returns popularity-sorted results.
Alternative: JustWatch API, OMDb, Letterboxd scraping. TMDB has the most comprehensive filter support for free.
sessionStorage
Stores movies and roomId so they survive page refresh within the same tab. Automatically cleared when the tab closes — no stale data left behind. Scoped per tab (two tabs = two sessions).
Alternative: localStorage (persists across sessions — bad, could join wrong room) or React state only (lost on refresh — bad).
Railway (backend)
nixpacks.toml auto-detects Node.js and runs npm start. Zero Docker needed. Provides persistent Postgres via DATABASE_URL. Auto-deploys from GitHub.
Alternative: Fly.io, Render. Railway was chosen for simplicity and the nixpacks zero-config deployment.
node-fetch
Used in tmdb.js for HTTP requests to TMDB. Node.js 18+ has native fetch, but older versions don't. The package is a polyfill that works across all versions.
Alternative: Native fetch (Node 18+), axios. node-fetch is minimal and API-compatible with browser fetch.

Design Patterns Used

Module Singleton

socket.js exports one shared socket instance. ES module caching ensures every import gets the same object.

Null Object (Optional DB)

pool = null when no DATABASE_URL. All callers check if (pool). App degrades to in-memory without any code changes.

Error as Value

joinRoom() returns { error } instead of throwing. Caller checks result.error. Server stays up even on bad input.

Ref Forwarding (Stale Closure Fix)

MovieCard uses useRef for voted and onVote so event listeners always access current values without re-registering.

Lazy State Initializer

Room.jsx uses useState(() => JSON.parse(...)) — function form runs once on mount, not on every render. Avoids repeated sessionStorage reads.

Deterministic Generation

MatchModal's CONFETTI array is computed at module load with index-based math instead of Math.random() — stable across re-renders and StrictMode double-calls.

Known Limitations

⚡
Server restart loses all rooms
The in-memory Map is wiped on restart. Postgres only stores movies, not votes — so players mid-game lose their progress. DATABASE_URL mitigates the "can't join" problem but not mid-session state loss.
👥
Hard 2-player limit
The room capacity is hard-coded to 2. Expanding to groups would require rethinking match detection (unanimous? majority?) and the done-tracking logic.
🎬
Fixed 20-movie pool
Movies are fetched once per room and don't change. If both players swipe through all 20 with few matches, they can't request more without creating a new room.
🔒
No authentication
Anyone with the room code can join. Rooms only support 2 players so the third person gets an error, but the first person to join with the code wins the slot.
📈
Linear scan for disconnect lookup
getRoomBySocket() iterates all rooms to find one by socket ID. With thousands of concurrent rooms, this O(n) scan becomes a bottleneck. Fix: maintain a reverse Map (socketId → roomId).
🌐
Single-server architecture
Two players must connect to the same server instance. Load-balanced multi-server deployment would break room synchronization (rooms are in-memory per process). Fix: shared state via Redis + Socket.io Redis adapter.
