The Room Data Structure

Every room is stored as this object in the rooms Map. Understanding this shape is essential to understanding every function below.

{ id: "ABCDEF", // 6-char code — redundant copy of the Map key users: ["socket_creator_id", "socket_joiner_id"], // max 2 socket IDs movies: [ // 20 movies from TMDB (same for both players) { id: 550, title: "Fight Club", poster: "...", year: "1999", rating: 8.4, overview: "..." }, ... ], votes: { // each player's votes, keyed by socket ID "socket_creator_id": { 550: true, // creator liked Fight Club 27205: false, // creator passed Inception }, "socket_joiner_id": { 550: true, // joiner also liked Fight Club → MATCH! } }, matches: [ // movies both players liked (populated by recordVote) { id: 550, title: "Fight Club", ... } ], done: Set { "socket_creator_id" } // players who finished all 20 movies }

Why this structure?

votes is a two-level object (socketId → movieId → boolean) rather than an array because the core operation is a O(1) lookup: "did socket X vote for movie Y?" With an array you'd need a linear scan every time. The current structure gives O(1) read and write for every vote check.

done is a Set (not an array) because Set.has() is O(1) and Set.add() is idempotent — adding the same socket twice doesn't create duplicates. This is safer than an array for the "player-done" event, which could theoretically be emitted more than once.

Error as Value Pattern

joinRoom returns either { room } or { error } instead of throwing exceptions. The caller pattern is:

const result = await joinRoom(code, socketId);
if (result.error) {
  socket.emit('error', result.error);
  return;
}
// result.room is safe to use here

Advantages: the error is a string that can be sent directly to the client, no try/catch boilerplate, and the server never crashes from expected failure modes (bad code, full room).

Match Detection: When Exactly Does It Fire?

The algorithm is event-driven, not polling. Every incoming vote triggers a cross-check. The question "has the other player already liked this movie?" is answered in O(1) using the votes object.

Two valid orderings both produce a match:

# Player A votes first, Player B votes second:
recordVote(room, A, movie550, true)  → other=B, votes[B][550] = undefined → no match yet
recordVote(room, B, movie550, true)  → other=A, votes[A][550] = true → MATCH!

# Player B votes first, Player A votes second:
recordVote(room, B, movie550, true)  → other=A, votes[A][550] = undefined → no match yet
recordVote(room, A, movie550, true)  → other=B, votes[B][550] = true → MATCH!

The check runs on the second like to arrive — whichever player votes later triggers the match detection.

exported markDone(roomId, socketId)
Called when a player finishes all 20 movies. Adds them to the done Set. Returns the accumulated matches ONLY when ALL players are in the done Set — otherwise returns null. This is the "gate" that controls when matches are revealed.
87export function markDone(roomId, socketId) {

89 if (!room) return null;Room doesn't exist (server restart). Return null — caller won't broadcast.
91 room.done.add(socketId);Set.add is idempotent — if a player emits player-done twice, done Set won't double-count them.
93 const allDone = room.users.length > 0 && room.users.every(id => room.done.has(id));every() returns true only if ALL users are in done. length > 0 guard prevents an edge case: if users is [] (both disconnected before finishing), every() on empty array returns true — which would incorrectly trigger all-done.
94 return allDone ? room.matches : null;Returns matches[] (possibly empty []) when done. null means "still waiting". Caller MUST check !== null (not just truthy) because [] is falsy.

The Critical !== null Check

In index.js, the caller is:

const matches = markDone(roomId, socket.id);
if (matches !== null) {
  io.to(roomId).emit('all-done', matches);
}

This must be !== null and NOT if (matches), because:

  • null → still waiting → don't broadcast
  • [] → all done, no matches → broadcast the empty array (players see "No matches this time")
  • [{...}] → all done, matches found → broadcast the array

If we used if (matches), an empty [] would be falsy and "all done, no matches" would never broadcast the all-done event — both players would wait forever.
