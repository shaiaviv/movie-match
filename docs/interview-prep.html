1. "Walk me through the project"

Memorize this verbal answer. It should take about 90 seconds and cover what, why, and how.

MovieMatch is a real-time two-player movie swiping app. The core problem it solves is the "what should we watch" deadlock — two people who can't agree. The mechanic is simple: both players see the same set of movies simultaneously and swipe right or left. When both like the same film, it's a match.

On the technical side, I built it with a Node.js backend using Express and Socket.io for real-time communication, and a React frontend with Vite. The movie data comes from TMDB's Discover API — I fetch three random pages of results for each room, shuffle them, and take the top 20. That variety means different rooms see different movies.

The interesting design decision is how matches are revealed. I don't broadcast a match the moment it's detected — instead, I hold all matches until both players finish swiping. This is intentional: if Player A finishes early and sees a match pop up, they know exactly which movies Player B liked. Holding the reveal preserves the mutual surprise and prevents information asymmetry.

State is managed server-side in a simple in-memory Map — each room is an object with a users array, a votes dictionary, and a matches array. I also added an optional Postgres fallback so rooms survive a server restart. The client persists movies in sessionStorage so page refreshes within the same tab work seamlessly.

2. Technology Questions

Q What is Socket.io and why did you use it instead of raw WebSockets? ▶

Socket.io is a real-time communication library that wraps WebSockets with several production-grade features. The key ones I relied on are:

  • Rooms: built-in concept of named groups. socket.join(roomId) and io.to(roomId).emit() let me broadcast to exactly the two players in a room without tracking socket IDs manually.
  • Automatic reconnection: if a player's connection drops briefly, Socket.io reconnects and fires the 'connect' event. I re-emit join-room on reconnect so the new socket ID gets registered with the room.
  • HTTP polling fallback: if WebSockets fail (corporate proxies, old browsers), Socket.io falls back to long-polling transparently.

Raw WebSockets would require implementing all of this manually. The trade-off is a larger dependency, but for a real-time multi-player app, Socket.io's abstractions are worth it.

TL;DR: Socket.io's room system, reconnection handling, and fallback transport made multi-player sync significantly easier than raw WebSockets.
Q Why is the socket a singleton module? Why not create it inside each component? ▶

JavaScript ES module imports are cached by the runtime. Every file that does import socket from '../socket.js' gets back the exact same object. This makes socket.js a natural singleton without any special design pattern code.

If I created the socket inside each component, I'd have multiple connections to the server (one per component), and events registered in one component wouldn't be visible to another. I'd also need React Context or a global store to share it.

The singleton means Home.jsx and Room.jsx both use the same socket, and navigating between them doesn't disconnect/reconnect — the socket stays alive across route changes.

TL;DR: ES module cache = automatic singleton. One socket, shared across all components, no Context needed.
Q Why is autoConnect: false set on the socket? ▶

When socket.js is first imported, the module runs immediately. If autoConnect were true, a WebSocket connection would be established the moment the app loads — before the user has done anything.

With autoConnect: false, the socket is created (the object exists) but doesn't connect. The actual connection happens in Home.jsx's useEffect when socket.connect() is called. This gives intentional control over when the connection happens, avoids a connection on app load if the user navigates away quickly, and is more predictable to reason about.

TL;DR: Prevents an automatic connection on import. The app explicitly connects when it's actually needed.
Q Why use sessionStorage instead of localStorage or React state for movies? ▶

Each option has a different scope:

  • React state: cleared on page refresh. If a player accidentally refreshes mid-game, they lose the movie list and the room breaks.
  • localStorage: persists until explicitly cleared. A player coming back the next day could accidentally "join" the old room with stale movies.
  • sessionStorage: scoped to the browser tab and the current page session. It survives a refresh (within the same tab) but is automatically cleared when the tab closes. Perfect for ephemeral game data.

Two different browser tabs also get their own sessionStorage — which is correct behavior since they're treated as two separate players.

TL;DR: sessionStorage survives refresh but auto-clears on tab close. Exactly the right lifetime for game session data.
Q What is TMDB and why did you choose it? ▶

TMDB (The Movie Database) is an open community-maintained movie database. Their API is free for non-commercial use with just a free account registration — no credit card, no billing, no rate limit concerns for small apps.

The specific reason I chose TMDB over alternatives (OMDb, JustWatch, IMDb) is the Discover API. It's a single endpoint that supports all the filters I needed — genre, year range, runtime, original language, US age certification, and vote average — as URL parameters. Other APIs would have required building my own filter logic on top of more limited data.

TL;DR: Free API with a Discover endpoint that natively supports all the filters we need in a single request.

3. Architecture Questions

Q How does match detection work? ▶

Match detection happens in recordVote() on the server. When a vote comes in, the function stores it in a nested object: room.votes[socketId][movieId] = liked.

For every liked = true vote, it immediately checks if the other player has already voted true for the same movie: room.votes[other]?.[movieId] === true. If both have liked it, the movie is pushed to room.matches[].

Crucially, this detection happens in real time as votes arrive, but the matches are not broadcast until both players finish. The markDone() function returns the accumulated matches array only when all users are in the done Set — that's when all-done is emitted to both players.

TL;DR: Detected greedily on each vote using a two-pointer check. Revealed only when all players finish (privacy gate).
Q Why doesn't the room creator have to "join" again? How does Room.jsx work for them? ▶

When Room.jsx mounts, it always emits join-room — even for the creator who is already in the room. The joinRoom() function handles this gracefully: it checks if (room.users.includes(socketId)) return { room } — returning success immediately without adding the user again.

This design serves two purposes: (1) It's a clean single-path for all players — Room.jsx doesn't need to know if the user is the creator or a joiner. (2) It handles reconnection after a server restart — even the creator needs to re-register their socket ID when the server restarts.

TL;DR: joinRoom() is idempotent — adding an existing socket is a no-op. Room.jsx always calls it, handling both creators and joiners with the same code path.
Q What happens if the server restarts while players are in a game? ▶

With only in-memory storage: the rooms Map is wiped. When Socket.io reconnects, Room.jsx re-emits join-room. The server checks the Map, doesn't find the room, and emits an error. Room.jsx catches this and redirects to the home page.

With DATABASE_URL configured: joinRoom() has a cache-miss fallback. If the room isn't in memory, it queries Postgres for the room's movies. It reconstructs a fresh room object with those movies (empty votes and matches). Both players can rejoin — but vote history is lost since Postgres only stores movies, not votes. They'd essentially start a fresh game in the same room.

TL;DR: Without DB: kicked to home on reconnect. With DB: room recovers its movie list but loses vote progress — clean restart in the same room.

4. Critical Thinking Questions

Q What are the limitations of this approach? ▶
  • Single-server only: in-memory rooms can't be shared across server instances. Horizontal scaling requires Redis + the Socket.io Redis adapter.
  • Linear scan on disconnect: getRoomBySocket() iterates all rooms. At thousands of concurrent rooms, this becomes a bottleneck. Fix: maintain a reverse Map.
  • No authentication: anyone with the code can join. A third person gets an error, but room codes are guessable given enough tries.
  • 20-movie hard cap: no pagination or "load more" within a session.
  • Solo playability: a single player can swipe through all movies, but can never see matches (the "all done" gate requires both players). The UX shows a permanent "waiting for partner" screen if the second player never joins.
Q How would you scale this to handle 10,000 concurrent rooms? ▶

The current single-server architecture breaks at scale for two reasons: memory limits and the inability to load-balance across instances (two players might hit different servers).

The fix involves three changes:

  • Redis for room state: move the rooms Map to Redis. Redis is shared across all server instances, so any instance can handle any socket event. Room data would serialize to JSON (same structure).
  • Socket.io Redis adapter: the io.to(roomId).emit() calls need to broadcast across all instances. The Redis adapter handles this by publishing to a Redis pub/sub channel.
  • Persistent vote storage: with horizontal scaling, you'd want to use a proper DB (Postgres) for votes too, not just movies — to handle mid-session server reassignments.
TL;DR: Redis for shared state + Socket.io Redis adapter for cross-instance broadcasting. This is the standard Socket.io scaling pattern.
Q What would you do differently with more time? ▶
  • Reverse socket-to-room Map: replace the O(n) getRoomBySocket() scan with a Map lookup. One line of change, big scalability win.
  • Persist votes to DB: currently only movies are persisted. Storing votes would allow mid-session recovery after server restart.
  • Spectator mode: let a third person watch the swiping session in real time without voting.
  • Movie streaming links: integrate JustWatch API to show where each matched movie can be streamed.
  • Room expiry via Postgres: currently cleanup happens in a setTimeout in memory. A proper Postgres TTL cleanup job would be more reliable.
Q How would you test this? ▶

Unit tests for roomManager.js — it's pure functions with no I/O (the pool calls are behind if(pool) guards). Test: createRoom generates valid codes, recordVote detects mutual likes correctly, markDone returns null until all players done, generateCode excludes ambiguous characters.

Integration tests for the socket event flow using a real Socket.io test client (socket.io-client in a test environment). Simulate two players connecting, creating a room, voting, and finishing. Assert that all-done is received with the correct matches.

E2E tests with Playwright — open two browser contexts, create and join a room, both swipe, verify the results screen shows matches.

The hardest part to test is timing — the 350ms setTimeout in MovieCard.exit. Use fake timers (vitest's vi.useFakeTimers) to control this.

5. Code Gotchas — "What does this line do?"

These specific lines from the codebase are the most likely to be asked about in a technical interview.

if (matches !== null) { io.to(roomId).emit('all-done', matches); }
Why !== null and not just if (matches)?
Because an empty array [] is falsy in the context of if (matches) — it would evaluate as false even though it means "everyone is done, just no matches." The markDone() function returns null to mean "still waiting" and [] or a populated array to mean "all done." Checking !== null correctly distinguishes these two cases.
if (other && room.votes[other]?.[movieId] === true) {
What does the ?. do here and why === true instead of just truthy?
Optional chaining (?.): if room.votes[other] is undefined (the other player hasn't voted yet, or their votes object doesn't exist), the expression short-circuits to undefined instead of throwing TypeError: Cannot read property of undefined.

=== true instead of truthy: undefined is falsy, but we need to distinguish "hasn't voted" (undefined) from "voted no" (false). Strict equality ensures only an explicit true vote triggers a match.
const socket = io(SERVER_URL, { autoConnect: false });
The module runs immediately — why isn't the socket connected after this line?
autoConnect: false creates the socket object and configures it but does NOT initiate the WebSocket handshake. The socket is in a "disconnected but ready" state. socket.connect() must be called explicitly to trigger the actual connection. This gives the application control over when the connection happens — in Home.jsx's useEffect, after the component mounts.
const [movies, setMovies] = useState(() => { try { return JSON.parse(sessionStorage.getItem('movies') || '[]'); } catch { return []; } });
Why pass a function to useState instead of the value directly?
This is the lazy initializer pattern. If you pass a value directly (useState(JSON.parse(...))), the expression is evaluated on every render — sessionStorage is read and JSON is parsed 10, 20, 100 times. Passing a function means React only calls it once (on mount) and caches the result. For expensive initialization like JSON.parse, this avoids unnecessary work on re-renders.
const onVoteRef = useRef(onVote); useEffect(() => { onVoteRef.current = onVote; }, [onVote]);
Why not just use onVote directly in the event listeners?
Event listeners registered inside useEffect(fn, []) close over the initial value of onVote and never update. If onVote changes (e.g. parent re-renders with a new function reference), the listener still calls the old one — a stale closure bug. The fix: store onVote in a ref. The ref object itself never changes (same reference), but ref.current is always updated to the latest value. Listeners read onVoteRef.current(liked) at call time, getting the current version.
const CONFETTI = Array.from({ length: 32 }, (_, i) => ({ left: `${(i * 3.23 + 3) % 94}%`, ... }));
Why is this defined at module level and not inside the component?
If this were inside the component body, React would regenerate 32 new random positions on every render. In React StrictMode (dev mode), components render twice — confetti positions would change between renders, causing a visual "jump". By defining CONFETTI at module level, it's computed exactly once when the file is first imported. Positions are fixed for the entire session. The deterministic math (i * 3.23) also means the same positions are reproduced predictably.
rooms.set(id, { votes: { [socketId]: {} }, done: new Set(), });
Why use an object for votes but a Set for done?
votes is an object keyed by socketId, then by movieId: { "socket123": { 550: true, 27205: false } }. It's a lookup structure — we need to find "has this socket voted for this movieId?" in O(1). An object/Map is the right choice.

done just needs to track membership: "is this socketId done?" Sets are optimized for membership checks (.has()) and prevent duplicates — if a player sends player-done twice, Set.add is idempotent (no duplicate entries).
return shuffle(formatMovies(combined)).slice(0, 20);
Walk me through what this line does step by step.
Three transformations in order:
  1. formatMovies(combined) — takes the raw TMDB objects array, filters out incomplete entries (no poster/title/overview), and maps each one to the app's lean shape: { id, title, overview, poster, year, rating }. Returns a filtered, reformatted array.
  2. shuffle(...) — Fisher-Yates in-place shuffle of the formatted array. Randomizes order so every room sees movies in a different sequence. Returns the same array (mutated).
  3. .slice(0, 20) — takes the first 20 elements. Three pages × 20 results = up to 60 movies; we only want 20. Since the array was shuffled, this picks a random 20 from the combined pool.
